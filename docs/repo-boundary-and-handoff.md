# Repo Boundary and Handoff Model (Framework vs Website)

## Decision
Use **two repos**:

1. `Elevate-Flow` (this repo): framework, agents, clerk-service, Pete pipelines, contracts.
2. `mission-control-dashboard` (web repo): UI only, presentation layer, auth/session UX.

Do **not** mix web UI code back into framework repo.

## Why this avoids confusion
- Clear ownership boundaries.
- Smaller PRs and faster reviews.
- Website deploy risk cannot break factory logic.
- Agent ops cadence stays in one repo (`Elevate-Flow`).

## Ownership
- JJ: operations cadence, acceptance checks, release go/no-go.
- Vlad: implementation in web repo.
- Pete: data/logic producer in framework repo.
- Coppa: security review on both repos for exposed surfaces.

## Source of Truth Rules
- Business and operating truth: `ZERO-MISSION-SOT.md` in `Elevate-Flow`.
- Quant logic truth: `projects/pete-dfs/*` in `Elevate-Flow`.
- UI truth: routes/components in `mission-control-dashboard`.
- API/data contract truth: versioned contract in `Elevate-Flow/docs/contracts/`.

## Integration Contract (v1)
Web repo only reads from:
- Clerk status:
  - `GET /health`
  - `GET /v1/status` (requires `X-MC-KEY`)
- Pete output artifact (generated by pipeline):
  - `best_bet`
  - `team_parlay`
  - `player_prop_parlay`
  - `logic_summary`
  - `goal_tracker`

No UI-side betting logic calculations.

## Branch Strategy
- Both repos:
  - `main` protected (no direct pushes).
  - feature branches: `codex/<scope>` or `vlad/<scope>`.
- Promote flow:
  1. Merge to repo branch
  2. JJ validation checklist
  3. Deploy

## Required Release Checklist
Before web deploy:
1. Pull latest `Elevate-Flow` branch.
2. Verify Pete pipeline tests pass.
3. Verify clerk `/health` and `/v1/status`.
4. Verify contract version unchanged or UI updated for new version.
5. Record decision in `/decisions`.

## Operational Guardrails
- No secrets in commits, docs, screenshots, or logs.
- `MC_API_KEY` never stored in repo.
- `X-MC-KEY` only injected via runtime env.
- If contract changes: version bump + migration note.

## Anti-Patterns (do not do)
- Copy/paste Pete logic into UI repo.
- Store daily raw data snapshots in Git.
- Cross-edit both repos in one mixed PR.
- Use submodules for this setup.
